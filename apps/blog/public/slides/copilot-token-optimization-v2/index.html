overview

Model choice matters most

Every request consumes AI Credits (1 credit = $0.01) based on input, output, and cached tokens — priced per model at published API rates.

TierExamplesWhat it costs
EfficientLightweight models (Haiku-class, mini models)Lowest per-token rates — cheap for scoped edits
FrontierClaude Sonnet 5, GPT-5.5e.g. Sonnet 5: $2 / 1M input · $10 / 1M output
Max & thinkingClaude Opus 4.7, extended thinking modesHighest rates — thinking tokens billed as output

Same prompt, different model — the bill can differ by an order of magnitude.

06

overview

The default: everything in one chat

Most people keep a single session open — plans, tool output, failed attempts, and corrections all stay in the same context window. Every new message re-sends all of it.

07

overview

Slightly smarter: start fresh

Instead of resteering inside a bloated thread, open a new chat with a clear directive. Same instructions and tools — without dragging the failed conversation forward.

08

overview

So… when should we start a new chat?

When it stops pushing back and just says you're right — that's your cue to reset.

There is a better way — split work into phases with fresh context and human review between each. The rest of this talk builds up to it, step by step.

Session info — where tokens went

  • 155K / 200K — 78% of context used in one session
  • Messages — 30% of the window is chat history
  • Tool results — 33% from reads, searches, and writes
  • Files — only 1.6% — your @file picks aren't always the culprit

Quality may decline as the limit nears — compact or start fresh before you hit the wall.

17

context window

The smart zone vs the dumb zone

Smart zone

Top of context — system instructions, tools, and your latest message. The model attends here best. Stay under ~40% used.

Dumb zone

Middle fills with chat history and tool output. Past the line, quality drops — wrong answers, missed details, "you're absolutely right."

18

key insight

↻

Everything is re-sent every turn

Chat history doesn't live on the server — it's re-sent with every message. Cached-token pricing softens the re-bill, but cost still compounds with thread length, and quality drops as the window fills.

01 · UnderstandDefine what & done before anything runs

Bad

"Fix the notification bug."

No scope, no success criteria, stale chat history. Expect 3–4 clarification rounds — each re-sends the full context window.

Good

"Here is PBI #4821 — read the Description and Acceptance Criteria. Ask clarifying questions before you change any code."

Requirements + AC define done upfront. Agent confirms understanding in ~1 cheap turn instead of guessing through code.

01 Be specific

New chat. Name the file, function, and expected behavior — don't carry unrelated history into the task.

02 Ask the agent to ask

"Before implementing, ask me clarifying questions." Or use a discovery / requirements skill to gather information deeply.

03 Use PBI Description & AC

Paste or @file the backlog item. Description + Acceptance Criteria tell the agent what "done" means before Gather or Execute.

02 · GatherInformation to have in your project

Maintain these docs up front — agents read them instead of exploring the whole repo. Cheaper gather, fewer wrong assumptions.

Project overview

What the product does, tech stack, key domains, and where to start. Keep it in README.md or docs/overview.md — link from copilot-instructions.md.

System architecture

Major components, data flow, integrations, and boundaries between services. One diagram + short prose beats the agent inferring structure from file search.

Code base summary

Orientation map so the agent knows where features live without scanning every folder.

Folder structure

Top-level layout — apps/, packages/, tests, configs — and what each area owns.

Available commands

Build, lint, test, dev, and deploy scripts from package.json or Makefile — so Verify and Gather use the right tooling.

02 · GatherUse subagents to gather

Bad

Main agent explores the whole repo in one thread — @file every module it finds, skills enabled, search results piling into context.

Every file read stacks into your context window — billed on every subsequent turn in the thread.

Good

Spawn an explore subagent: "Map the notification flow — return paths, patterns, and integration points only. No full files." Feed its summary into Plan & Execute.

Gather cost stays in the subagent's isolated window. Main thread stays lean for planning and coding.

01 Delegate to an explore subagent

Offload codebase exploration to an explore / research subagent — not the main thread. Instruct it to return compressed summaries only: file paths, patterns, and integration points — not full file dumps. Feed that distilled report into Plan & Execute so the main agent plans and codes without wandering.

04 · ExecuteScope the work & right-size the model

Bad

Claude Sonnet: "Rewrite the entire notification feature module in one shot."

200+ lines of output billed at frontier output-token rates — for work an efficient model handles in a scoped edit.

Good

Efficient model: "Add retryCount to deliver() per the plan." Frontier only if multi-system debugging is needed.

Small, focused output. Right model for the job — negligible quality loss, large cost difference.

01 Scope each edit

One function, one file, one change. A whole-module rewrite in one prompt balloons output tokens and multiplies rework.

02 Match model to task

Use an efficient model for scoped edits; reserve frontier models for multi-system debugging or ambiguous design trade-offs.

03 Execute per the plan

Implement what Plan agreed on — don't reopen design decisions mid-thread. Each detour re-sends the full context window.

05 · VerifyAlways verify with project commands

Bad

"Does this look right?" — you eyeball the diff, merge, and find out later in CI that lint or tests fail.

No automated check means bugs surface downstream. Full execute loop repeats with more context and regenerated output.

Good

"Before finishing, run npm run lint, npm test -- NotificationService, and npm run build. Fix any failures."

Agent catches issues in one verify pass. Fix in place — cheaper than restarting the thread after CI breaks.

npm run build npm run lint npm test npm run typecheck npm run test:regression npm run e2e

04 · Optimize Tips

Optimize tips

If every turn re-sends everything, the highest-ROI move is shrinking what gets re-sent — and remembering that a wrong answer is the most expensive token of all.

What can you cut without hurting answer quality?

54

optimize tips

The re-send checklist

You've seen each of these already — here they are as one checklist.

  • Focused prompts — ask specific questions; don't paste open files, Copilot already has them
  • Lean instructions — keep copilot-instructions.md to 10–20 lines; load only task-relevant skills
  • Deliberate @file — five large files for one question multiplies every later turn
  • Fresh chat per task — resetting costs nothing; stale history costs every turn
  • Right-size the model — Sonnet to rename a variable is a senior engineer booking a meeting room
55

optimize tips

Good architecture speeds discovery

DDD, event-driven design, and vertical slices give the agent clearer boundaries — fewer files to hunt, stronger signals in context, less explore cost.

DDD

Bounded contexts and ubiquitous language tell the agent where a feature lives — and what not to touch.

Event-driven

Explicit events and handlers make flows discoverable without tracing every call site by hand.

Vertical slices

Feature-shaped folders keep related code together — one path to read instead of a layered scavenger hunt.

optimize tips

Quality is a cost control

Vague or wrong answers trigger follow-ups, more context, and regenerated code — the redo loop dwarfs any per-token saving. The cheapest interaction is the one that gets it right the first time.

One planning session in Plan mode typically costs less than a single failed implementation attempt.
