10

01 Understand

Define what & done before anything runs

Bad

"Fix the notification bug."

No scope, no success criteria, stale chat history. Expect 3–4 clarification rounds — each re-sends the full context window.

Good

"Here is PBI #4821 — read the Description and Acceptance Criteria. Ask clarifying questions before you change any code."

Requirements + AC define done upfront. Agent confirms understanding in ~1 cheap turn instead of guessing through code.

01 Be specific

New chat. Name the file, function, and expected behavior — don't carry unrelated history into the task.

02 Ask the agent to ask

"Before implementing, ask me clarifying questions." Or use a discovery / requirements skill to gather information deeply.

03 Use PBI Description & AC

Paste or @file the backlog item. Description + Acceptance Criteria tell the agent what "done" means before Gather or Execute.

02 Gather

What the agent is trying to find

Every task triggers a search for context. These six areas drive exploration — and every byte loaded is billed on every turn.

Where

Target files, folders, entry points — which code owns this feature?

Patterns

How similar work is built here — components, hooks, services, naming.

Project rules

copilot-instructions.md, lint/format config, repo conventions.

Dependencies

Imports, shared types, API contracts, callers of the code you touch.

Tests

Existing test files, fixtures, mocks — what already covers this area?

Tooling & skills

Build, lint, test commands — plus framework docs and skills for this task.

12

02 Gather

Information to have in your project

Maintain these docs up front — agents read them instead of exploring the whole repo. Cheaper gather, fewer wrong assumptions.

Project overview

What the product does, tech stack, key domains, and where to start. Keep it in README.md or docs/overview.md — link from copilot-instructions.md.

System architecture

Major components, data flow, integrations, and boundaries between services. One diagram + short prose beats the agent inferring structure from file search.

Code base summary

Orientation map so the agent knows where features live without scanning every folder.

Folder structure

Top-level layout — apps/, packages/, tests, configs — and what each area owns.

Available commands

Build, lint, test, dev, and deploy scripts from package.json or Makefile — so Verify and Gather use the right tooling.

02 Gather

Use subagents to gather

Bad

Main agent explores the whole repo in one thread — @file every module it finds, skills enabled, search results piling into context.

Every file read stacks into your context window — billed on every subsequent turn in the thread.

Good

Spawn an explore subagent: "Map the notification flow — return paths, patterns, and integration points only. No full files." Feed its summary into Plan & Execute.

Gather cost stays in the subagent's isolated window. Main thread stays lean for planning and coding.

01 Delegate to an explore subagent

Offload codebase exploration to an explore / research subagent — not the main thread. Instruct it to return compressed summaries only: file paths, patterns, and integration points — not full file dumps. Feed that distilled report into Plan & Execute so the main agent plans and codes without wandering.

03 Plan

Resolve assumptions before coding

Bad

"Implement a notification system with email, in-app delivery, retry logic, and unread counts."

Model guesses data model and integration points. You redo parts 2–3 times — each pass regenerates thousands of output tokens.

Good

Plan mode: "What are the key design decisions? What should the data model look like? What are the integration points?"

~400-token plan, you correct assumptions, then one implementation pass. Cheaper than one failed attempt.

04 Execute

Scope the work & right-size the model

Bad

Claude Sonnet: "Rewrite the entire notification feature module in one shot."

200+ lines of output at 3–5× output token cost, plus premium request multiplier — for work a base model handles in a scoped edit.

Good

Base model: "Add retryCount to deliver() per the plan." Premium only if multi-system debugging is needed.

Small, focused output. Right model for the job — negligible quality loss, large cost difference.

01 Scope each edit

One function, one file, one change. A whole-module rewrite in one prompt balloons output tokens and multiplies rework.

02 Match model to task

Use the base model for scoped edits; reserve premium for multi-system debugging or ambiguous design trade-offs.

03 Execute per the plan

Implement what Plan agreed on — don't reopen design decisions mid-thread. Each detour re-sends the full context window.

04 Split phases across sessions

Don't combine implementation, tests, and refactor in one chat. Start a fresh session for each phase so context stays lean and you're not paying to re-send test logs and diffs on every turn.

18

05 Verify

Always verify with project commands

Bad

"Does this look right?" — you eyeball the diff, merge, and find out later in CI that lint or tests fail.

No automated check means bugs surface downstream. Full execute loop repeats with more context and regenerated output.

Good

"Before finishing, run npm run lint, npm test -- NotificationService, and npm run build. Fix any failures."

Agent catches issues in one verify pass. Fix in place — cheaper than restarting the thread after CI breaks.

npm run build npm run lint npm test npm run typecheck npm run test:regression npm run e2e

Rule: Always ask the agent to verify using your app's scripts — build, lint, test, regression — not visual review alone.

19

key insight

↻

Everything is re-sent every turn

Chat history doesn't accumulate cheaply on the server — it's serialized and transmitted with each new message. A 10-message thread can cost ~10× a single-message thread.

20

optimize

Reduce prompt size

  • Don't paste open files — Copilot already has them
  • Ask specific questions, not restated background
  • Start a new chat when you switch tasks — no cost to reset
21

optimize

Instructions & skills

  • copilot-instructions.md is included in every request — keep it 10–20 lines
  • Load only skills relevant to the current task
  • Use @file deliberately — five large files for one question multiplies cost

optimize

Match model to task

Base model

  • Autocomplete & inline suggestions
  • Renames & docstrings
  • Quick syntax questions

Premium model

  • Multi-system debugging
  • API & schema design
  • Large unfamiliar codebases

Sonnet to rename a variable = hiring a senior engineer to book a meeting room.

23

quality

Bad output costs more

Vague or wrong answers trigger follow-ups, more context, and regenerated code. The cheapest interaction is one that gets it right the first time.

quality

Plan before you implement

"Implement a notification system with email, in-app delivery, retry logic, and unread counts."

→ Model assumes. You redo. Thousands of tokens wasted per iteration.

One planning session in Plan mode typically costs less than a single failed implementation attempt.

summary

Four deliberate habits

  • Focused prompts — say less, mean more
  • Clean context — new chats, lean instructions
  • Model-appropriate tasks — right tool, right job
  • Plan before coding — resolve assumptions first
